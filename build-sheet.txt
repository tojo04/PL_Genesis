Here’s a practical build sheet for Cerebrum.

It is scoped for a hackathon MVP and aligned with the Neurotech track requirement to use at least one sponsor API/SDK, with strong fit to secure neural data, sovereignty, and privacy-preserving collaboration.

Cerebrum build sheet
1. Project goal

Build a web app where a user can:

connect wallet
upload a neural-data JSON file
encrypt it
store it on Storacha/Filecoin
grant access to another wallet
revoke access later
show authorized access succeeds and revoked access fails

That is the full MVP.

3. Core user stories
Owner
I connect my wallet
I upload my neural dataset
I store it securely
I grant access to a researcher
I revoke access later
Researcher
I connect my wallet
I request or attempt access
If approved, I can decrypt and view the dataset
If revoked, I can no longer access it

6. Data model
Neural dataset metadata
type NeuralDataset = {
  id: string
  ownerWallet: string
  cid: string
  fileName: string
  uploadedAt: string
  encrypted: boolean
  accessList: AccessGrant[]
}
Access grant
type AccessGrant = {
  wallet: string
  purpose?: string
  expiresAt?: string
  status: "active" | "revoked"
}
Mock neural session
type NeuralSession = {
  sessionId: string
  timestamp: string
  alphaBand: number
  betaBand: number
  thetaBand: number
  focusScore: number

10. Build order
Step 1: Project setup
create Next.js app
install Tailwind
install shadcn/ui
create routes
make basic layout and nav
Step 2: Wallet connection
set up FCL
connect Flow Wallet
show connected address in navbar and dashboard

Done when: user can connect and see wallet address.

Step 3: Mock data and upload UI
make upload form
accept .json
parse and preview fields
validate structure

Done when: a sample JSON file previews correctly.

Step 4: Encryption
encrypt uploaded JSON in browser
output encrypted blob/string

Done when: raw file becomes encrypted before storage.

Step 5: Storacha/Filecoin upload
upload encrypted payload
return CID
show CID in UI
store dataset metadata in local state or simple DB

Done when: encrypted file is stored and CID is visible.

Step 6: Lit access control
create access condition tied to allowed wallet
protect encryption key
store access configuration with dataset

Done when: dataset has owner and allowed wallet policy.

Step 7: Consent manager
dataset list
add wallet to access list
optional purpose + expiry
revoke existing wallet

Done when: owner can grant and revoke through UI.

Step 8: Access test page
connect as second wallet
select dataset
attempt decrypt
if authorized, show parsed neural data
if unauthorized, show denied state

Done when: one wallet succeeds, another fails.

Step 9: Revocation test
owner revokes access
researcher retries
decryption fails

Done when: revoked access clearly stops working.

Step 10: Polish
better UI cards
badges like Active / Revoked
success/error toasts
short explainer text for judges

12. Acceptance criteria

You are MVP-complete when all of these work:

owner connects wallet
owner uploads neural JSON
file is encrypted before storage
encrypted file is stored and CID is shown
owner grants access to another wallet
approved wallet decrypts successfully
owner revokes access
approved wallet can no longer decrypt

If these work cleanly, you already have a strong hackathon demo.

13. Demo script build checklist
Scene 1

Show problem:
“Neural data is highly sensitive and users need revocable, sovereign control.”

Scene 2

Connect owner wallet.

Scene 3

Upload simulated neural dataset.

Scene 4

Show encrypted upload to Storacha/Filecoin and returned CID.

Scene 5

Grant researcher wallet access.

Scene 6

Switch to researcher wallet and decrypt successfully.

Scene 7

Switch back, revoke access.

Scene 8

Researcher tries again and gets denied.

That is your story arc.

14. Risks to avoid
spending too long on Flow smart contracts
trying to add live EEG hardware
building a full backend unnecessarily
overengineering permission logic
adding ML before MVP works

Keep the core demo working first.
